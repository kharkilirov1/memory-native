# MN-26 — образ memory-native модели 2026 года (blueprint)

Как сложились бы наши проверенные компоненты + актуальные архитектурные приёмы '25–'26 в одну
модель, **если довести внедрение M-рычагов до конца**. Каждый блок помечен по правилу «witness или
не считается»:

- ✅ **доказано** — есть реальный свидетель в репо (тест/прогон), число держится.
- 🔧 **внедрение-ждёт** — модуль реализован и проверен в изоляции, но НЕ вшит в обучаемую GPT.
- 📐 **проекция** — обоснованная инженерная ставка, своего числа пока нет (нужен kernel/scale/железо).

Это не отчёт о готовом — это целевая сборка. Сегодня в реальную модель внедрён только **базовый
counter-синапс**; всё остальное ниже — дорожная карта в одной картинке.

---

## 1. Тезис дизайна

**Memory-native, не compute-native.** Вес, оптимизатор и градиент живут в sub-byte конечном
автомате; активации — O(1) через обратимость. Выигрыш модели — **память и архитектурные рычаги**, а
не скорость шага (метод bandwidth-bound; «меньше бит» не ускоряет GEMM — см. `PERF_ANATOMY.md`).
Поэтому 2026-сборка ставит не на «быстрее матмул», а на: *больше ёмкости на тот же активный compute*
(MoE/MoD/memory), *плотнее сигнал на токен* (MTP), *почти нулевое состояние оптимизатора*.

## 2. Стек по слоям

```
            tokens ─► embedding (counter-linear, tied/untied)
                          │
        ┌─────────────────┴───────────────── reversible coupling (O(1) активации) ✅
        │  x1,x2  ─►  y1 = x1 + F(x2);  y2 = x2 + G(y1)
        │
        │   F = ATTENTION sub-block
        │       • GQA + RoPE                                   📐 (стандарт '24+, не наш)
        │       • QKV/out = counter-linear (ternary, 0.75B/w)  ✅ базовый метод
        │       • опц. MLA-сжатие KV-кэша (DeepSeek-класс)     📐
        │
        │   G = FFN sub-block
        │       • Counter-MoE (M4): N экспертов, top-k,        ✅ доказан (бьёт dense)
        │         load-balance aux; эксперт = counter-FFN      🔧 внедрение-ждёт
        │
        ├─ Mixture-of-Depths (M10): top-k токенов на блок,     🔧 реализован / число confounded
        │   capacity<1 → лёгкие токены пропускают блок
        │
        └─ anchors каждые A блоков (recompute-from-anchor)     ✅ exact, parity A∈{1,2,3,5,8}

            ... × n_layer ...
                          │
          ┌───────────────┴─────────── головы
          │  next-token  (основная)
          └  Multi-Token Prediction (M9): +k голов, untied     🔧 реализован / toy-fail, нужен scale
```

## 3. Весовой субстрат (фундамент — ✅ внедрён уже сегодня)

| свойство | значение | статус |
|---|---|---|
| persistent state | **0.75 байта/вес** (ternary t + 6-бит counter, 4 кода/3 байта) | ✅ `packed.py` |
| оптимизатор | **в состоянии** — нет FP-мастера, нет Adam-моментов (≈0 доп. памяти) | ✅ `counter.py` |
| адаптивность | RMS row-stat (дешёвый аналог Adam-variance) | ✅ |
| base-update частота | **slow-fast (M3): −8…32×** обновлений базы без потери точности | 🔧 внедрение-ждёт |
| DDP | состояния **бит-в-бит** между рангами (hash-SR безусловно, torch-SR при RNG-симметрии) | ✅ `test_ddp_*` |

## 4. Compute-пути (Tensor Cores)

| путь | выигрыш | статус |
|---|---|---|
| int8 forward (`Y=XTᵀ`, per-token scale) | **×2.05** изолированно (d=2048), 0.95× dense на d≥768 | ✅ `ACCELERATION.md` |
| int8 update (переиспользует сохранённую int8-активацию) | **×1.45 (K=2048) … ×2.16 (K=8192)** vs fp32 cuBLAS | ✅ |
| fp8 grad_w (#5, e4m3, `_scaled_mm`) | parity-gated; ~4% rel-err, error-feedback поглощает | 🔧 реф готов / ядро Blackwell |
| NVFP4-веса в GEMM (int4-условие) | оправдан **только на Blackwell** (на T4 Amdahl убивает) | 📐 `INT4_WGRAD.md` |

## 5. Kernel-fusion (где живёт реальная скорость — 📐, ядра не написаны)

- **#2 ternary-prologue**: decode в mainloop GEMM (forward зависит только от видимого `t`) — ✅ инвариант подтверждён, ядро 📐.
- **#1 lagged-epilogue**: counter-update внутри grad_w GEMM из аккумулятора (lagged → per-element) — ✅ энейблер подтверждён, ядро 📐.
- **#4 persistent flip-patch cache**: декод амортизированно ≈0 — ✅ bit-exact в PyTorch уже.
- **#3 overlap** recompute↔update — 📐 GPU-gated.

## 6. Что реально получаешь (бюджет, честно)

**Память (главный приз — ✅):**
- веса 0.75 Б/вес vs 2 Б (fp16) → **~2.7× меньше** на весах;
- оптимизатор ≈0 vs Adam 8 Б/вес (m+v fp32) → **на оптимизаторе экономия почти 100%**;
- активации O(1) (reversible) vs O(L) → линия по глубине срезается;
- итого тренировочный peak радикально ниже dense+AdamW при тех же параметрах.

**Скорость (честно — НЕ приз сегодня):**
- полный шаг **0.89–0.95× dense** на T4 (чуть медленнее); путь к ≥1× — int8 fwd+update на масштабе
  + fused-ядра (#1/#2), которых пока нет.

**Качество/ёмкость (рычаги — смешанно):**
- ✅ **M4 Counter-MoE** — бьёт dense при равном активном compute (E=16: 1.632 vs 1.655), монотонный scaling.
- 🔧 M1 memory-FFN — на реальных данных слабее dense (+12%); M4 закрывает нишу лучше.
- 🔧 M9 MTP / M10 MoD — реализованы, но без масштабного/причинно-корректного свидетеля.

## 7. Bill of materials — что внедрено / что осталось

**Внедрено в реальную GPT за флагами (✅, `test_model_integration.py`):** полезные M-рычаги теперь
выбираются конфигом, dense-путь бит-в-бит не тронут.

| # | компонент | вердикт | как включить | статус |
|---|---|---|---|---|

| 2 | **M3 slow-fast** линейки | ✅ PASS | `GPT(kind="slowfast", rank=, merge_every=)` | **ВНЕДРЕНО** |
| 3 | **M1 memory-FFN** | 🔧 слабее dense | `GPTConfig(ffn="memory", ffn_cells, ffn_k)` | **ВНЕДРЕНО** (compute-saver) |
| 4 | **M2 2:4 group** | 🔧 PARTIAL | `GPT(kind="group", group=, keep=)` | **ВНЕДРЕНО** |

**Намеренно НЕ вшито (с причиной):**

| компонент | почему не вшито |
|---|---|
| **M10 MoD** | в causal-LM ломает причинность: top-k собирает токены через позиции/батч и attention среди них смешивает будущее с прошлым (известный confound). Сначала причинно-безопасный witness, потом роутинг. |
| **M9 MTP** | это **отдельная модель** `MTPGPT` (не сабслой Block); toy-fail без untied heads на scale. Уже существует как самостоятельный модуль. |
| fused-ядра #1/#2, NVFP4 | 📐 kernel/Blackwell-работа, не PyTorch-уровень. |

## 8. Одной строкой

**MN-26 = обратимый GPT, где каждая линейка — counter-синапс (0.75 Б/вес, оптимизатор-в-состоянии),
FFN — Counter-MoE (M4, доказанный выигрыш), глубина — MoD, цель — MTP, compute — int8/fp8 Tensor
Cores.** Приз — **память и ёмкость-на-compute**, не скорость шага. Сегодня внедрён фундамент; первый
реальный шаг сборки — вшить **M4 за флагом `ffn="moe"`**.
